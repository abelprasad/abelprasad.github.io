Finding a good internship posting is a needle-in-a-haystack problem. Job boards are noisy, most listings are irrelevant, and by the time you've filtered through fifty results you've already lost an hour. I got tired of doing this manually and built Scout — a self-hosted agent that does it for me every morning.

## The Architecture

Scout runs as a set of cooperating services rather than one monolithic script. An LLM planner handles goal decomposition — figuring out what to search for, which boards to hit, and how to sequence the scraping runs. Playwright handles the actual scraping, spinning up a headless browser to pull postings from sites that block raw HTTP requests. FastAPI ties everything together as a lightweight API layer, and SQLite stores every posting it finds.

The multi-agent structure wasn't overengineering. The scraping logic and the scoring logic are genuinely separate problems with different failure modes, and separating them made both easier to iterate on.

## The Scoring Engine

The first version of Scout was almost useless. It pulled fifty-plus postings a day with no filtering, which meant I still had to manually read through everything. The scoring engine was the fix.

Each posting gets scored against a set of role keywords (things I actually want — "software engineer", "backend", "AI/ML") and a set of negative filters (things I don't — "senior", "10+ years", "PhD required"). The score is a weighted sum, and only postings above a threshold make it into the daily digest.

The negative filters cut the noise more than anything else. Going from fifty noisy results to a tight shortlist of eight to twelve relevant postings was the moment Scout actually became useful.

## The Daily Digest

Every morning, Scout sends a ranked HTML email and a Telegram message with each posting's score, match rationale, and direct apply link. The email is generated by the FastAPI backend and sent via SMTP. The Telegram notification uses the Bot API.

The ranked format matters. The top result on any given morning is worth reading immediately. The bottom of the list is for slow mornings. Having the rationale alongside the score means I can trust the ranking — I know *why* Scout surfaced something, not just that it did.

## What I'd Do Differently

The SQLite schema grew organically and it shows. I'd design the tables more carefully upfront if I were starting over. I'd also add a web UI for reviewing the shortlist — the email digest is good but a simple dashboard with one-click flagging would make the weekly review faster.

Scout is one of those projects I actually use every day, which makes it worth keeping maintained. Every time I tweak the scoring weights or add a new board, I can see the impact immediately in the next morning's digest.
